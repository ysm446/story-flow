# goals.md — プロジェクトの目的と完成形

作成日時: 2026-07-08 16:39
更新日時: 2026-07-10 18:59

## 何を作るか

**story-flow** — 短編ストーリー生成・鑑賞デスクトップアプリ。

作者が事前に「シーンカード」（メディア + 短い指示文 = ブリーフ）を大量に用意しておき、
始点・終点（と任意の中間点）を置くと、ローカル LLM が在庫カードから間を埋め、
左から 1 シーンずつ清書して一本の物語に仕立て、それを鑑賞する。
`lm-graph` をフォークして構築する。

詳細仕様は [docs/spec.md](../spec.md) が唯一の正。本ファイルはその要約と価値基準。

## 面白さの核（最重要価値）

- **リプレイ性**: 同じ素材から毎回すこしずつ違う短編が生まれること。これが生命線。
- カードは完成した本文ではなく**作者の意図（ブリーフ）**。清書は全編 LLM が書き、
  読み味が毎回変わる。ただしブリーフの意図・確定事実は壊さない。
- 一貫性: 逐次生成 + StoryState（確定事実）の持ち越しで「3 枚目の赤い傘が
  5 枚目で消える」型のドリフトを防ぐ。

## 完成形

### v1（まずここを完成させる）

- **Vault**: カード CRUD、メディア登録、タグ/ロール、埋め込み計算、検索、在庫密度可視化 — フル実装
- **Compose**: React Flow で始点・中間・終点アンカーを置いて繋ぐ（順序 = 並びのみ）
- **Generate**: 逐次清書 + StoryState 持ち越し（穴埋めなし、FIXED スロットのみ）
- **Theater**: Ken Burns + オート送り + クロスフェードで鑑賞

### v1.5（本命のパズル型）

- `fill_gap`: 候補検索（sqlite-vec + FTS + ロール）→ LLM 選択の二段で穴を自動で埋める
- 多様性制御、浅いバックトラック、target_tone の引力、在庫密度チューニング

### v2（構想）

- 引力（アトラクタ）方式の分岐・複数終端、鑑賞後の軌跡マップ、TTS + BGM、VN 的選択分岐
- **ゲームブック派生（アイデアメモ）**: フローチャートが機能したら、ゲーム的コンテンツへ派生させる。
  「分岐点の StoryState を起点に、次シーン候補を LLM に N 本書かせる → ユーザーが選ぶ →
  選んだ分岐の StoryState を次へ持ち越す」という、逐次清書パイプラインの分岐版。
  - v1.5 の「候補検索 → LLM 選択」の二段構えを「分岐候補の生成」に転用できる。
  - **一括生成禁止に触れないため、分岐は必ず「一手ずつ」に限定する**（木構造を一気に吐かせない）。
  - これを潰さない前提: Generate を状態機械として 4 相分離しておく／StoryState を
    「分岐点のスナップショット」として復元可能な形で持てるようにしておく。
  - 位置づけはあくまで「一本道が閉じた後の派生先」。v1 完成が最優先で、いまは実装しない。
- **設定資料 RAG（アイデアメモ）**: 物語・キャラクター・世界観の背景をテキストファイルで持ち、
  清書時に RAG として参照して深みを出す。作者が書く恒久的な canon（= series bible）。
  - **Phase 1 実装済み（2026-07-10）**: 「背景設定」として作品（workspace）ごとにタイトル付き
    複数メモを保存し、清書時に**全文注入**（workspaces.lore、Compose 生成設定から編集）。
    検索（retrieve → inject）は未実装。メモが長文化して窮屈になったら以下の RAG 方式に進化させる。
    保存単位も当初構想のライブラリ共通ではなく作品ごとを採用（作者選択）。
  - **StoryState（生成中に確定した動的な事実）とは別レイヤ**として扱う。両方 writer に食わせるが、
    StoryState = 一話限りの発生事実、背景テキスト = 全話・全生成をまたぐ不変の設定、と役割を分ける。
  - パイプラインへの乗せ方: 清書時に「そのシーンのブリーフ + 現在の StoryState」を手がかりに
    背景テキストから関連チャンクを検索し、writer コンテキストへ注入（retrieve → inject）。
    既存の sqlite-vec + FTS5 ハイブリッド検索をそのまま流用できる。
  - **spec 判断ブロックとの線引き**（壊さないための必須条件）:
    - 「埋め込み・検索の対象は常にカード」に対し、背景テキストは**別コーパスとして明確に分離**する
      （カード検索の結果に混ぜない。参照専用の資料棚として扱う）。
    - 「候補検索 → LLM 選択の二段」は _カード選択_ の話。背景 RAG は選択ではなく**文脈拡張**なので
      別物として扱い、全在庫を LLM に見せる話にはしない。
    - 「清書結果は index しない（一方向）」は維持。背景テキストは入力側の資料であって清書結果ではない。
  - 置き場所: ライブラリフォルダ（`prompts.json` と同じ層）に `lore/` のようなテキスト置き場を足す。
  - 位置づけは v1.5 の充実枠（`fill_gap` と並ぶエンリッチ機能）か v2。v1 の最小経路には入れない。

## 重視する価値・判断基準

1. 迷ったら「一本道 → 逐次清書 → 鑑賞」の最小経路を優先し、多機能化しない。
2. spec.md の「### 判断」ブロックは確定事項。実装時に蒸し返さない。
3. ML コンポーネントは再実装せず、subprocess / HTTP でラップする。
4. データの流れは `Vault → Compose → Generate → Theater` の一方向。
   清書結果は保存するが index しない（`DB → 物語` の一方向、逆流させない）。

## 非目標

- 一括生成（全シーンをまとめて LLM に書かせること）は採らない。
- v1 の段階で v1.5 / v2 の機能を実装しない（差し込める設計にはする）。
- 完全なビームサーチ等の重い探索は v1.5 でも実装しない。
