# 時系列の先読み（シーンのつながり改善）— 設計メモ

作成日時: 2026-07-10 18:45
更新日時: 2026-07-10 18:45

状態: **アイデア段階（未実装）**。2026-07-10 の作者コメントから起票。

## 背景・課題

現状の writer（`backend/services/writer.py`）が各シーンの清書時に見ているのは:

- 物語全体のプロット（任意）
- StoryState（**過去**に確定した事実: 人物・持ち物・出来事・場所・時間・トーン）
- 今回のカードのブリーフ + 位置ラベル（導入 / 途中 / 結末）+ 追加指示

つまり「過去」は StoryState 経由で知っているが、**「未来」は結末の位置とプロット以外
まったく知らない**。次のシーンにどんなカードが来るかを知らずに書くため、
「次のシーンは夜の駅なのに、今のシーンを昼の室内で締めてしまい、つなぎが跳ぶ」
といった時間・場所の飛びが構造的に起こりうる。

作者コメント（2026-07-10）:
「LLM が事前にカードの時系列を予測していれば、前後関係がおおよそ分かって
自然なつながりになるのでは」

## 方式案

### 案 A: 次シーンの予告（小さい変更）

- 各シーンの清書プロンプトに「次のシーンの予定: <次カードのブリーフ 1〜2 行>」を添える。
  writer は次への布石（時間・場所・視線の受け渡し）を打てるようになる。
- pipeline はカード列（slots）を最初から持っているため、`_build_user_prompt` に
  1 項目足すだけ。逐次生成の原則はそのまま。
- 注意: 次のブリーフの内容を先取りして書きすぎるリスクがあるため、
  「次の内容は書かない。つなぎの準備だけ」という指示を必ず添える。
- おまかせスロット（gap）が次に来る場合はカード未確定のため予告できない
  （「次は未定」と伝える、または省略）。

### 案 B: 構成メモ（時系列アウトラインの事前パス）

- 生成開始時に全カードのブリーフ列を LLM に 1 回見せ、「各シーンのおおよその時系列・
  経過時間・場面転換」を短いメモとして出力させる。以後の各シーン清書に
  そのメモを定数として添える。
- 全体の時間の流れ（同日か、数年後か等）が最初に固定されるため、つながりはより安定する。
- コスト・リスク: 生成のたびに LLM 呼び出しが 1 回増える。メモが外れていると
  逆に全シーンを縛る。おまかせスロット確定前はそのシーンの内容が読めない。

### spec との整合

spec の確定判断「生成は逐次のみ・一括生成禁止」は**清書**についての判断であり、
A・B とも抵触しない（B は清書ではなく事前の構成パス）。実装時は spec §6 に追記する。

## UI 方針（作者指示 2026-07-10）

**ひとつの方式に決め打ちしない。設定のプルダウンで方式を選べるようにする**:

- なし（現状のまま。既定値）
- 次シーンの予告（案 A）
- 構成メモ（案 B）

保存先は他の生成系設定と同様（UI 設定 or ワークスペース単位かは実装時に判断。
シーンの長さは workspace 保存なので、物語の性質に紐づくならワークスペース側が有力）。

## 実装順の目安

まず案 A（変更が小さく、逐次生成の哲学=「左の結果を見て右を書く」に馴染む）で
効果を確認し、物足りなければ案 B を追加する。UI は最初からプルダウンにしておき、
案 B は選択肢を後から足す。
