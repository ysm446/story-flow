# 中間カード採用（おまかせスロット / fill_gap）アルゴリズム

作成日時: 2026-07-10 04:15
更新日時: 2026-07-11 00:00

Compose の「おまかせスロット（？ノード）」を、生成時に在庫カードで埋める仕組みの設計資料。
spec §1.5 / §6.2 の確定判断「候補検索 → LLM 選択の二段」の実装。
実装: `backend/services/selection.py`（選定本体）、`backend/services/pipeline.py`（逐次ループ）、
`src/phases/compose/ComposePhase.tsx`（おまかせノード）、`src/lib/chain.ts`（スロット導出）。

## 設計原則（spec の確定判断）

- **全在庫を LLM に見せない**。ベクトル検索で k 件（既定 6）に絞ってから 1 枚選ばせる。
- **左から 1 枚ずつ確定**する。穴が連続しても、選んだカードを直前カード（A）に繰り込んで
  から次の穴を解く。選択と清書は同じ逐次ループ上（一括生成禁止と同じ理由）。
- **ロールはハードフィルタではなくスコアボーナス**（2026-07-09 作者懸念への対応）。
  rising / turn の境界が曖昧でも候補から漏れず、role 未設定カードは汎用として扱われる。
- 選ばれたシーンは `is_fixed=0` + `selection_reason`（選定理由）付きで保存し、
  作者があとから「なぜこれが選ばれたか」を確認できる。

## UI（おまかせノード方式。2026-07-10 作者選択）

- Compose のアセットエリア「＋ おまかせスロット」で破線の「？」ノードを配置し、
  アンカーと同様に線で繋ぐ。**1 ノード = 1 枚**。
- ノードのプロパティで **希望ロール**（任意）・**追加指示**・**BGM 指名** を指定できる。
- workspace graph には `kind: "gap"` + `target_role` で保存（ノード ID は `gap-` 接頭辞。
  kind 省略 = card の後方互換）。
- 制約: **始点・終点はカード必須**（穴は固定点の「間」を埋めるもの）。
  部分再生成（テイクからの撮り直し）は確定済みカード列に対してのみで、
  穴の引き直しは「新しいテイクを生成」で行う。

## 選定フロー（穴 1 つごと、`fill_gap`）

```
在庫（未使用カード）が空？ ──yes→ エラーで生成停止（明示メッセージ）
   │no
[第一段] retrieve_candidates: 候補 k=6 件に絞る
   クエリベクトル = A（直前に確定したカード）と B（次の固定アンカー）の
                    保存済み埋め込み（card_vec）の内分点 (1-t)·A + t·B
     t = 1/(次の固定アンカーまでの残り穴数+1)。単独の穴は t=1/2（中点）。
     A〇〇〇B では 1 つ目の穴が t=1/4（A 寄り）、確定が進むごとに 1/3, 1/2 と
     B 寄りへ滑る（2026-07-11 追加。従来は常に中点で、穴が連続すると
     毎回ほぼ同じあたりを検索してしまっていた）
     │ 片方しか無い → その 1 本を使う
     │ 両方無い → A/B の brief + tone_so_far + target_tone の合成テキストを埋め込む
     │ 埋め込みサーバも不可 → ロール一致優先 + ランダムの候補に劣化（リプレイ性は保つ）
   sqlite-vec KNN で一次取得（プール 48 件）
     → 使用済み（used_ids）を除外
     → target_role 一致カードは距離 −0.15（ROLE_BONUS。ボーナスであってフィルタではない）
     → スコア順に k 件。埋め込み未計算カードで枠が余れば劣化候補で補充
   （候補が 1 枚だけならその場で確定し、LLM は回さない）
[第二段] select_card: 候補だけを LLM に提示して 1 枚選ばせる
   入力: プロット / StoryState / A の要旨 / B の要旨 / 求める条件（希望ロール・目標トーン）
         / 候補一覧（ID・タイトル・ロール・タグ・ブリーフ。清書はさせない）
   出力: {"card_id": ..., "reason": ...}（temperature 0.6。選択に幅 = リプレイ性）
     │ LLM 失敗 → 検索スコア最上位の候補に劣化（生成は止めない）
     │ 候補外の ID → 同上
確定 → SSE で selected（カード + 理由）を通知 → そのまま writer で清書
     → used_ids に追加、prev_card を更新して次のスロットへ
```

### A と B の決め方

- **A（prev_card）**: 直前に確定したカード。固定アンカーでも、直前の穴で選ばれた
  カードでもよい（選んだそばから繰り込む）。先頭に穴は置けないので A が無いのは
  部分再生成のコピー区間明けのみ（その場合も state は引き継がれている）。
- **B（next_anchor）**: そのスロットより右で最初に現れる**固定アンカー**。
  穴が連続する場合、間の穴はすべて同じ B に向かって橋を架ける。

### SSE イベント

| イベント | 意味 |
|---|---|
| `selecting` | 候補検索 + LLM 選択中（Generate ノードに「カード選定中…」） |
| `selected` | カード確定（card_id / card_title / reason。ノードがカード表示に変わる） |
| `scene` | 清書完了（is_fixed=false + selection_reason 付き） |

## 劣化マトリクス

| 障害 | 挙動 |
|---|---|
| 在庫ゼロ | エラーで停止（「カードを追加するかスロットを減らして」） |
| A/B の埋め込み未計算 | 合成テキストの埋め込みで代替 |
| 埋め込みサーバ停止 | ロール一致優先 + ランダムの候補に劣化 |
| LLM 停止・出力不正 | 検索スコア最上位の候補を採用（理由にその旨を記録） |
| LLM が候補外 ID を返す | 同上 |

## 定数・チューニング点（`selection.py`）

| 定数 | 値 | 意味 |
|---|---|---|
| CANDIDATE_K | 6 | LLM に見せる候補数（spec §14 で確定） |
| ROLE_BONUS | 0.15 | ロール一致の距離ボーナス（L2 正規化ベクトルの距離スケール前提） |
| SELECTOR_TEMPERATURE | 0.6 | 候補内選択のブレ幅（リプレイ性の源泉のひとつ） |
| _KNN_POOL | 48 | ベクトル一次取得の件数（除外・加点の母集団） |

selector の system prompt は `backend/prompts/selector.md` が既定で、writer と同じ
プリセット機構（設定 UI）で編集できる。出力形式指示はシステム側が必ず付与する。
エンドポイントは `STORY_FLOW_SELECTOR_URL` で writer と分離可能（既定は同一サーバ）。

## 残課題（spec §7 のチューニング 3 論点）

- **多様性**: used_ids 除外と temperature は実装済み。target_tone のサイコロ
  （引力をランダムにずらす）は未実装。実測で同じカードに偏るようなら
  「直近テイクで選ばれたカードへのペナルティ」も候補。
- **行き止まり**: 浅いバックトラック（1 手戻して除外して再選択）は未実装。
  現状は候補が尽きると劣化選択で進む。B に繋がらない選択の検出自体が今後の課題。
- **在庫密度**: `/vault/stats` の可視化で作者側が把握する（アルゴリズムでは解決しない）。
